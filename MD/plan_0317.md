## Phase 1: Transactional Outbox Pattern

### 목표
예약 성공 시 알림을 반드시 전송하되, 알림 실패가 예약을 롤백시키지 않도록 보장

### 구조
```
T1 (예약 트랜잭션):
  INSERT booking
  INSERT outbox_event   ← 알림 로직 없음, 이벤트 기록만
COMMIT                  ← 원자적 보장

T2 (OutboxPollingPublisher, @Scheduled 5초마다):
  SELECT outbox_event WHERE status = 'PENDING'
  → notificationService.createNotification()
  → UPDATE outbox_event SET status = 'PROCESSED'
```

### 신규 파일

**1. `domain/OutboxEvent.java`**
```
- id (PK, AUTO_INCREMENT)
- aggregateType: String  (예: "BOOKING")
- aggregateId: Long      (예: bookingId)
- eventType: String      (예: "BOOKING_CREATED")
- payload: TEXT          (JSON)
- status: Enum           (PENDING / PROCESSED / FAILED)
- createdAt: LocalDateTime
- processedAt: LocalDateTime
```

**2. `repository/OutboxRepository.java`**
```
- findByStatus(OutboxEventStatus status)
- findByStatusAndCreatedAtBefore(...)  ← 재처리 쿼리
```

**3. `service/OutboxPollingPublisher.java`**
```
@Scheduled(fixedDelay = 5000)
@Transactional(REQUIRES_NEW)  ← 독립 트랜잭션
- PENDING 이벤트 조회
- eventType 분기 → notificationService 호출
- status = PROCESSED 업데이트
- 실패 시 status = FAILED (재처리 대상)
```

### 수정 파일

**4. `service/BookingService.java`**
```
createBooking() 내부:
  bookingRepository.save(booking)  // 기존
  outboxRepository.save(           // 추가
    new OutboxEvent("BOOKING", booking.getId(), "BOOKING_CREATED", payload)
  )
```

**5. `scripts/init.sql`** (또는 JPA create-drop으로 자동 생성)
```sql
CREATE TABLE outbox_event (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  aggregate_type VARCHAR(50) NOT NULL,
  aggregate_id BIGINT NOT NULL,
  event_type VARCHAR(100) NOT NULL,
  payload TEXT,
  status VARCHAR(20) NOT NULL DEFAULT 'PENDING',
  created_at DATETIME NOT NULL,
  processed_at DATETIME,
  INDEX idx_outbox_status (status, created_at)
);
```

### After 테스트 결과 (실측)
- Case 2 (알림 포함): 37ms avg, 32~51ms 분산 (1.6x)
- Case 3 (Outbox 실제 INSERT): **36ms avg, 33~40ms 분산 (1.2x)**
- BEFORE_MEASUREMENT_REPORT.md After 섹션 추가 완료

### 완료 기준
- [x] `OutboxEvent` 엔티티 및 저장 동작 확인
- [x] `OutboxPollingPublisher` 5초 주기 실행 로그 확인
- [x] 예약 생성 후 알림 DB 저장 확인 (T1 커밋 → T2 처리)
- [x] After 타이밍 측정 완료 (Case 3: 36ms, 분산 1.2x)

## Phase 2: MySQL Spatial Index (R-Tree)

### 목표
`findAll() + Java Haversine` → `ST_Within + SPATIAL INDEX`로 교체
EXPLAIN type: `ALL` → `range`, 스캔 행수 96.2% 감소 (Before 측정에서 이미 증명됨)

### 수정 파일

**1. `domain/ParkingSpace.java`**
```
BEFORE:
  private Double latitude;
  private Double longitude;

AFTER:
  @Column(columnDefinition = "POINT SRID 4326")
  private org.locationtech.jts.geom.Point location;

  // 하위 호환을 위한 편의 메서드
  public double getLatitude()  { return location.getY(); }
  public double getLongitude() { return location.getX(); }
```

의존성 추가: `hibernate-spatial` (Hibernate 6 이상 기본 포함 여부 확인 필요)

**2. `repository/ParkingSpaceRepository.java`**
```java
@Query(value = """
    SELECT * FROM parking_space
    WHERE ST_Within(location, ST_Buffer(ST_GeomFromText(:wkt, 4326), :radius))
    ORDER BY ST_Distance_Sphere(location, ST_GeomFromText(:center, 4326)) ASC
    """, nativeQuery = true)
List<ParkingSpace> findWithinRadius(
    @Param("wkt") String centerWkt,
    @Param("center") String centerWkt2,
    @Param("radius") double radiusDegrees
);
```

**3. `service/ParkingSpaceService.java`**
```
searchByLocation():
  BEFORE: findAll() → Java stream filter (Haversine)
  AFTER:  findWithinRadius(lat, lng, radius) → DB가 처리
```

**4. `scripts/init.sql`**
```sql
-- ParkingSpace 테이블에 SPATIAL INDEX 추가
ALTER TABLE parking_space
  ADD COLUMN location POINT SRID 4326 NOT NULL,
  ADD SPATIAL INDEX idx_location (location);
```

**5. `dto/ParkingSpaceCreateRequestDto.java`**
- latitude/longitude 필드 유지 → 저장 시 POINT로 변환하는 로직 추가

### After 측정 결과
- EXPLAIN type: `ALL` → `range`, key: `NULL` → `idx_location`
- 스캔 행수: 9,880 → 378 (96.2% 감소), filtered: 1.23% → 100%
- 실행 시간: 3.30ms → 2.42ms (27% 단축)

### 완료 기준
- [x] `EXPLAIN type = range`, `key = idx_location` 확인
- [x] `ParkingSpace` POINT 컬럼 + `GeoCacheService` Write-through
- [x] 위치 기반 검색 API (Redis GEO → MySQL Spatial fallback)

## Phase 3: Resilience4j + Graceful Shutdown

### 목표
외부 호출(Redis, 알림) 장애 시 예약 서비스 보호

### 구현 항목

**CircuitBreaker**
- 적용 대상: `DistributedLockUtil.executeWithLock()`, `OutboxPollingPublisher`
- 설정: 실패율 50% 초과 시 Open, 30초 후 Half-Open

**Bulkhead**
- 적용 대상: `BookingService.createBooking()`
- 동시 처리 스레드 제한 (예: 20개)

**Graceful Shutdown**
```yaml
server.shutdown: graceful
spring.lifecycle.timeout-per-shutdown-phase: 30s
```

## Phase 4: Redis GEO Cache Layer

### 목표
MySQL Spatial(쓰기/정확성) + Redis GEO(읽기/속도) CQRS 구조

### 구현 항목
- `GeoCacheService`: `GEOADD`, `GEORADIUS` 명령 사용
- Cache warm-up: 앱 시작 시 MySQL → Redis GEO 동기화
- Write-through: ParkingSpace 등록 시 MySQL + Redis 동시 갱신
- 읽기: Redis GEO 우선 → miss 시 MySQL fallback
